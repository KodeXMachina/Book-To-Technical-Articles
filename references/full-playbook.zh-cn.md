# 书籍解析为中文技术文章的方法规范

[中文](full-playbook.zh-cn.md) | [English](full-playbook.en.md)

> 这是原始中文 playbook。当前语言中立/双语化的主动规则以 `output-contract.md`、`workflow.md`、`writing-style.md`、`validation.md` 等拆分 reference 为准。

这份规范来自一次完整的 CUDA Programming Guide 教材化过程。它记录的不是某一本书的目录安排，而是一套之后可以复用的做法：把一本技术书从 PDF、图片、代码、章节结构，整理成适合人阅读、也能对照原书查漏的中文技术文章体系。

核心目标只有一句话：**不是把书机械转成 Markdown，而是把书重写成读者能建立模型、能写代码、能查细节、能反查原书覆盖的技术材料。**

## 1. 最终产物应该长什么样

一套合格的书籍解析结果应该至少包含四层材料：

| 层级 | 产物 | 作用 |
| --- | --- | --- |
| 原书解析层 | `markdown_output/chapters/*.md`、图片、代码块 | 保存可核对的原始结构，不追求可读性 |
| 阅读主线层 | `course_materials/v3/<module>/<article>.md` | 让读者真正读懂，承担技术文章职责 |
| Reference 补全层 | reference patch 或附录文章 | 承接 API、宏、枚举、限制、表格、弱覆盖项 |
| 审计层 | `source_coverage_audit.md` | 记录与原书章节、图片、代码、token 的覆盖差异 |

不要把这几层混成一份文件。原书解析层要忠实；阅读主线层要好读；reference 层要查全；审计层要诚实。

如果一本书需要更强的逐页溯源、逐图/逐代码提取、问答回填和后续博客派生，可以启用本文第 7 节的 **严格知识库模式**。它是默认流程的增强模式，不替代“可读技术文章 + reference patch + 覆盖审计”的主体系。

## 2. 基本判断

### 2.1 不写教案

不要把文章写成“学习目标、教学活动、课后练习”的教案。读者打开文章，是为了理解一个技术问题，而不是看教师如何授课。

更好的开头方式是：

- 这个问题在真实工程中为什么会出现。
- 如果不理解它，会写出什么错误代码。
- 本文将建立什么模型。
- 读完后能做出什么工程判断。

### 2.2 不机械翻译

不要逐段翻译 PDF，也不要把原书段落打散堆成中文条目。机械翻译的覆盖感很强，但阅读体验差，也不能真正帮助读者建立高层结构。

正确做法是：

- 保留原书知识点和术语。
- 重组表达顺序，让读者按理解路径前进。
- 对概念补上下文、动机、误区和工程判断。
- 在文末补 reference 区，承接不适合进入主叙事的大量 API 细节。

### 2.3 不只写摘要

600 页技术书不能压成几千行“总结”。摘要适合快速了解，不适合掌握。主线文章必须足够展开，reference patch 必须能覆盖细节。

合理结构是：

- 先按主题拆成多个模块。
- 每个模块再拆成多篇文章。
- 每篇文章围绕一个工程问题展开。
- 对密集章节额外生成 reference patch。

### 2.4 不让 supplement 永久游离

补充说明可以作为过渡，但最终阅读路径不应该长期依赖“双轨结构”。更好的状态是：

- 主线文章负责阅读。
- reference patch 挂在索引中，作为查阅入口。
- 审计报告明确说明哪些内容由主线覆盖，哪些由 patch 覆盖。

## 3. 推荐工作流

### 阶段 1：建立原书解析层

目标是把 PDF 解析成可核对的 Markdown，而不是马上写教材。

需要产出：

- 按章节切分的 Markdown。
- 图片资源目录。
- 代码块尽量保留。
- 表格尽量保留。
- 章节编号、页码、标题可追溯。
- 严格模式下额外生成逐页知识库，所有关键知识点带出处。

注意事项：

- 图片提取要检查透明通道、黑底、裁切错误和重复图。
- 图片文件名应稳定，最好包含章节和页码。
- 原书 front matter、版权、商标、索引类章节可按需求排除。
- 严格模式下使用统一出处格式：正文 `(P3)`，图片 `(P5-Fig2)`，表格 `(P6-Table1)`，代码 `(P7-Code1)`。
- 如果 OCR、图片、表格或上下文存在歧义，使用 `（待确认：...）` 标记，不要用猜测覆盖不确定性。
- 图片抽取时单边尺寸不超过 `2000px`；按实际内容自动缩放 DPI，避免生成过大文件。
- 严格模式下图片命名使用 `序号_简短描述.png`，描述来自图注或上下文，清理为合法文件名，长度不超过 40 字符。
- 如果图片或表格无法自动解析，保留图片链接，并标注 `（图表：P×-Fig×，内容待人工补充）`。

### 阶段 2：做顶层课程/模块规划

先不要急着写正文。应该先读目录、章节和内容密度，给出模块化方案。

模块划分原则：

| 原则 | 说明 |
| --- | --- |
| 按读者理解路径 | 不机械照搬原书章节 |
| 按工程主题聚合 | 把相关章节合并成一个主题模块 |
| 密集主题拆细 | API、语言、数值、互操作类章节需要拆成多篇 |
| 保持映射可查 | 每个模块要标明对应原书章节 |
| 不迷信固定数量 | 6 个课程可能太粗，12 个模块也不是上限 |

规划文件应包含：

- 模块标题。
- 对应原书章节。
- 为什么这样切。
- 每个模块应解决的问题。
- 后续文章清单。
- 哪些章节不纳入正文。

### 阶段 3：生成第一版阅读文章

第一版不要追求 reference 完整度，先追求读得通。

每篇文章建议骨架：

```markdown
# 用一个工程问题打开标题

## 1. 这个问题为什么会出现

## 2. 核心模型

## 3. 主线讲解

## 4. 代码路径

## 5. 工程判断表

## 6. 原书覆盖补全区

## 7. 读完应该形成的能力
```

其中：

- `核心模型` 优先使用 Mermaid 或表格。
- `代码路径` 要解释输入、输出、执行侧、同步边界和错误边界。
- `工程判断表` 用来回答“什么时候用 A，什么时候不用”。
- `原书覆盖补全区` 用来承接 API、宏、限制、图表和章节缺口。

### 阶段 4：做源书覆盖审计

写完第一版后，必须反查原书。不要凭感觉说“覆盖了”。

审计对象：

| 对象 | 检查方式 |
| --- | --- |
| 章节 | 原书每章是否进入阅读路径 |
| 图片 | 原书图片是否被引用并解释 |
| 代码 | 关键代码是否迁入、改写或解释 |
| API/token | 函数、宏、枚举、环境变量、编译选项是否保留 |
| 表格 | 大表是否完整保留、压缩重写或明确不复刻 |
| 高风险章节 | API、互操作、语言扩展、数值函数、系统章节重点检查 |

审计报告要诚实区分：

| 覆盖状态 | 含义 |
| --- | --- |
| 强覆盖 | 主线解释充分，关键细节也保留 |
| 中等覆盖 | 主题讲清楚，但 API 或表格不完整 |
| 弱覆盖 | 只提到概念，没有足够细节支撑 |
| 未覆盖 | 阅读路径中找不到对应内容 |
| 不纳入 | front matter、版权、商标、无教学价值内容 |

### 阶段 5：生成 reference patch

审计后不要推倒重写主线文章。对弱覆盖和未覆盖项，生成定向 reference patch。

reference patch 的定位：

- 不承担主叙事。
- 不写成散乱清单。
- 用“技术文章 + 逐项补全”的形式写。
- 把 API、宏、枚举、环境变量、限制、图片、表格压缩整理进去。

reference patch 每篇必须包含：

- 对应原文章和原书章节。
- 为什么单独补充。
- 按原书顺序或工程逻辑展开的正文。
- 弱覆盖/未覆盖知识点逐项补全。
- API、宏、环境变量、编译选项速查表。
- 原书代码/图片/表格补充说明。
- 读完后补上的能力。

### 阶段 6：更新索引和审计结论

生成 patch 后，要更新总索引和审计报告。

索引中应明确：

- 主线文章路径。
- reference patch 路径。
- 每篇 patch 补什么。
- 推荐阅读顺序。

审计报告中应更新：

- 文件数、行数、代码块数、图片覆盖数、表格行数。
- 剩余缺口。
- 哪些缺口已关闭。
- 哪些差异是有意不复刻。

## 4. 技术文章的写作风格

### 4.1 文章要像写给工程读者

读者不是来背概念的，而是要理解问题、写代码、排查 bug、做取舍。

好的表达方式：

- “这个机制解决什么工程问题。”
- “如果不用它，会遇到什么边界。”
- “它和相邻机制有什么区别。”
- “代码中的同步点在哪里。”
- “这个 API 的返回不代表工作已经完成。”
- “这个配置是提示，不是强制保证。”

避免的表达方式：

- “本节学习……”
- “教学目标……”
- “本段主旨……”
- “原文核对……”
- “下面介绍若干知识点……”

### 4.2 开篇要有问题感

不要用百科式定义开头。优先从真实场景切入。

例如：

- 错误：`CUDA Stream 是 CUDA 中的异步执行机制。`
- 更好：`写完第一个 kernel 后，你很快会遇到一个问题：CPU 已经提交了工作，但 GPU 还没完成，下一批数据又要进来。Stream 的价值就是把这些必须有序和可以重叠的工作表达清楚。`

### 4.3 概念要落到边界

技术书里的概念通常不是孤立知识点，而是边界：

| 概念 | 应该讲清的边界 |
| --- | --- |
| Stream | host 返回、GPU 执行、同 stream 顺序、跨 stream 无序 |
| Event | 记录点、等待点、计时边界 |
| Unified Memory | 地址统一、迁移、prefetch、advice、一致性 |
| Graph | 定义、实例化、更新、发射、capture 禁止项 |
| Atomic | 操作、memory order、thread scope |
| Shared memory | 可见性、生命周期、bank conflict |
| Driver API | context、module、function、launch、版本边界 |

### 4.4 代码要解释执行路径

代码不能只贴出来。每段核心代码至少说明：

- 这段代码运行在 host 还是 device。
- 输入来自哪里。
- 输出写到哪里。
- 哪些 API 是异步的。
- 哪个调用提供同步边界。
- 错误可能在哪里暴露。
- 哪些参数必须和硬件能力匹配。

严格知识库模式下，每个代码片段还必须补齐：

| 字段 | 说明 |
| --- | --- |
| 功能 | 这段代码解决什么问题 |
| 关键语句解析 | 逐条解释关键语句、API 调用和参数 |
| 输入/输出 | 输入数据、输出数据、状态变化 |
| 依赖 API/库 | 涉及的函数、库、宏、编译选项 |
| 原始出处 | 使用 `(P7-Code1)` 或多出处标注 |

如果原书把同一段代码拆成声明、初始化、实现、调用等多段展示，知识库中要还原成完整代码，并注明各段来源，例如“声明部分见 `(P5-Code1)`，实现部分见 `(P7-Code2)`”。如果原书存在优化前/优化后、不同实现方式、不同 API 路径的代码对比，需要生成 `代码对比` 表，说明实现差异、优劣势、性能特点和适用场景。

### 4.5 表格要服务判断

表格不是为了显得整齐，而是帮助读者做判断。

推荐表格类型：

| 类型 | 用途 |
| --- | --- |
| 决策表 | 什么时候用 A/B/C |
| 生命周期表 | 创建、使用、同步、销毁 |
| API 速查表 | 函数、宏、枚举、字段 |
| 错误排查表 | 现象、可能原因、检查点 |
| 对照表 | Runtime vs Driver、IPC vs VMM、pageable vs pinned |

### 4.6 Mermaid 用来表达结构

Mermaid 适合表达：

- 平台栈。
- 数据流。
- 生命周期。
- stream/event 依赖。
- memory migration。
- graph 实例化和更新。
- VMM reserve-map-access-release。

不要为了装饰画图。图必须回答一个结构问题。

## 5. 图片处理规范

原书图片不能只“贴上去”。每张图都应该有中文读图说明。

图片处理原则：

| 原则 | 说明 |
| --- | --- |
| 关键图必须保留 | 架构图、流程图、时间线、性能示意图不能丢 |
| 图片链接要稳定 | 使用相对路径，不使用临时路径 |
| 图片要配解释 | 说明读者应该看什么，不只是翻译 caption |
| 黑底/裁切要检查 | PDF 抽图可能出现透明通道或黑色底部问题 |
| 不强行插无意义图 | 商标、封面、版权图可不进入正文 |

读图说明的写法：

- 这张图展示了什么结构。
- 图中哪个箭头或区域最重要。
- 它和本文模型有什么关系。
- 它容易被误读成什么。

## 6. API/reference 内容处理规范

技术书中的 reference 内容很容易破坏文章可读性。处理方式是：主线少量嵌入，文末集中补全。

### 6.1 主线中只放理解必要项

例如讲 CUDA Graph 主线时，主文需要讲：

- 为什么有 Graph。
- capture 和显式构图区别。
- instantiate 和 launch 的关系。
- update 为什么有限制。

不应该在主叙事里列完所有 node API。

### 6.2 补全区列完整族

在 `原书覆盖补全区` 或 reference patch 中，应该尽量列全：

- API family。
- 宏。
- 枚举。
- 结构体字段。
- 环境变量。
- 编译选项。
- 限制条件。
- 版本差异。

### 6.3 大表格可以压缩，但要说明

compute capability、数学函数、C++ proposal、DXGI format 这类大表，不一定逐项复刻原书。

合理做法：

- 对常用项列压缩表。
- 对全族项按分类列出。
- 对不复刻的表明确说明“保留查表方法，不逐项复制原表”。
- 对高风险 P0 区域尽量列全。

## 7. 严格知识库模式

严格知识库模式适用于这类任务：用户明确要求逐页提取、强溯源、交互式问答补全，或者希望先建立一份结构化知识库，再从知识库派生技术文章或博客。它比默认流程更重，但能最大限度降低遗漏。

它不是默认流程的替代品。默认流程仍然是：原书解析层保存事实，阅读主线层负责可读，reference patch 负责查全，审计层负责对照原书。严格知识库模式只是在原书解析层和后续交互阶段增加更细的约束。

### 7.1 适用场景

| 场景 | 是否建议启用 |
| --- | --- |
| 标准技术书，需要一个月左右系统学习 | 可不启用，默认 v3 技术文章流程更合适 |
| API/reference 密度高，需要逐项查漏 | 建议启用 |
| 用户后续会围绕知识点频繁追问 | 建议启用 |
| 文档较短，但要求完整知识库 | 建议启用 |
| 只想生成一篇可读文章 | 不建议启用完整严格模式，可只借用博客派生规则 |

### 7.2 严格知识库产物

启用后，额外生成一份知识库 Markdown，标题使用 `知识库：[文档名称]`。推荐结构：

| 区块 | 内容 |
| --- | --- |
| 领域概述 | 核心问题、应用场景、读者应建立的总体模型 |
| 主题分层 | 从领域到主题、子主题、具体知识点的树状结构 |
| 代码/配置汇总 | 按功能归类的代码片段、配置、命令和 API |
| 图表索引 | 图片、表格、流程图、架构图及其出处 |
| 未明确点 | 所有 `（待确认：...）` 内容集中列出 |
| Q&A | 用户确认后的典型问答回填 |
| 更新记录 | 每次知识库变更的摘要 |

知识库不是博客，也不是最终技术文章。它更像事实仓库：强调完整、可查、可回溯。后续主线文章和博客可以从这份知识库中派生。

### 7.3 逐页提取与出处标注

严格模式下，先逐页处理，再做主题重组。逐页处理时不得因为后面会重写文章而丢掉细节。

出处格式统一如下：

| 内容类型 | 格式 | 示例 |
| --- | --- | --- |
| 正文知识点 | `(P页码)` | `(P3)` |
| 图片 | `(P页码-Fig序号)` | `(P5-Fig2)` |
| 表格 | `(P页码-Table序号)` | `(P6-Table1)` |
| 代码 | `(P页码-Code序号)` | `(P7-Code1)` |
| 跨页内容 | 多个出处并列 | `(P5-Code1, P7-Code2)` |

如果内容模糊、OCR 不可靠、图表含义无法自动确认，必须保留不确定性，例如：`（待确认：图中虚线箭头含义可能表示异步依赖，但原图注未说明）`。

### 7.4 图片、表格和流程图处理

图片抽取规则：

- 单个维度宽或高不超过 `2000px`。
- 按图片实际内容自动缩放 DPI，避免空白边和超大文件。
- 文件命名使用全局递增序号和简短描述，例如 `01_thread-hierarchy.png`。
- 描述优先来自图注；没有图注时取附近上下文；描述清理为合法文件名，长度不超过 40 字符。
- 每张进入正文或知识库的图都要配中文读图说明。

表格处理规则：

- 能结构化的表格转成 Markdown 表。
- 表格太大时可压缩，但要说明压缩原则。
- 无法可靠解析的图片表格保留原图，并标注 `（图表：P×-Fig×，内容待人工补充）`。

流程图和架构图处理规则：

- 如果原图结构清晰，保留图片并配读图说明。
- 如果更适合重画，使用 Mermaid 表达逻辑。
- Mermaid 只用于表达结构、数据流、生命周期或依赖关系，不做装饰。

### 7.5 代码解析与代码对比

严格模式下，代码片段使用固定信息块描述：

| 字段 | 内容 |
| --- | --- |
| 代码片段 | 功能名称和出处 |
| 功能 | 这段代码完成的任务 |
| 关键语句解析 | 逐条解释关键语句、API、参数和边界 |
| 输入/输出 | 输入、输出、状态变化、同步结果 |
| 依赖 API/库 | 函数、宏、库、编译选项、运行环境 |
| 注意事项 | 错误边界、性能边界、版本或平台限制 |

如果同一段代码被原书拆开，需要在知识库中合并还原为完整代码，并在说明中列出所有原始出处。不要因为代码跨页就只保留其中一段。

如果原书存在代码对比，必须生成对比表：

| 对比项 | 版本 A | 版本 B |
| --- | --- | --- |
| 实现方式 | 描述实现路径 | 描述实现路径 |
| 关键 API | 列出 API | 列出 API |
| 性能特点 | 描述吞吐、延迟、内存或同步差异 | 描述吞吐、延迟、内存或同步差异 |
| 适用场景 | 适合何种条件 | 适合何种条件 |
| 风险 | 可能出错的位置 | 可能出错的位置 |

### 7.6 交互式知识补全

严格模式下，用户后续提问时按这个规则处理：

- 先基于原书解析层、知识库和已生成文章回答。
- 如果文档未提及，明确写：`文档未提及，建议补充：...`。
- 可以补充简单示例、伪代码、Mermaid 或对比表，但要标明哪些是原书内容，哪些是扩展解释。
- 用户确认答案后，把结论回填到知识库对应章节。
- 知识库新增 `Q&A` 区块，记录典型问答、问题出处、回答摘要和回填位置。
- 每次更新知识库时，在 `更新记录` 区块追加一条摘要，例如：`2026-05-14：已补充 2.1.1 的算法边界和 P7 示例解释。`

“每次回复先明确当前阶段”只适用于严格知识库交互模式，不作为所有书籍处理任务的硬规则。阶段名可以是：`第一阶段：知识提取进行中`、`第二阶段：知识补全进行中`、`第三阶段：文章派生进行中`。

### 7.7 博客或单篇文章派生

博客风格输出是派生产物，不替代默认的 v3 技术文章体系。它适合在知识库和主线文章稳定后，为读者生成一篇更轻量的入口文章。

派生规则：

- 标题要有问题感或收益感。
- 开头写导读，说明文章提炼自哪份知识库或哪本书。
- 语言可以更通俗，但不能牺牲技术准确性。
- 可以加入强相关扩展知识，但不超过总字数的 20%。
- 扩展内容必须使用 `【扩展】` 或引用块标识。
- 保留核心知识引用，例如“参见 2.1.1”或“参见 `(P7-Code1)`”。
- 结尾做总结，不再引入大量新概念。

博客结构可采用：

| 区块 | 内容 |
| --- | --- |
| 导读 | 本文解决什么问题，来自哪份材料 |
| 为什么需要这个技术 | 用工程场景打开 |
| 核心概念解析 | 用比喻、图、表降低理解成本 |
| 实战示例 | 放最小代码或流程 |
| 进阶要点 | 陷阱、性能、版本、平台 |
| 总结 | 回到关键模型和实践判断 |

### 7.8 分批交付规则

不要把“超过 20 页必须停下询问是否继续”作为默认规则。默认应尽量端到端完成一轮处理。

只有在以下情况才分批：

- 用户明确要求按章节或页数分批输出。
- 单次上下文无法安全容纳完整输出。
- PDF 解析质量很差，需要人工逐批确认。
- 严格知识库模式下，某一批存在大量 `（待确认：...）`，继续处理会放大错误。

分批时，每批结束要说明已完成范围、待确认点和下一批内容，但不要无意义地频繁打断。

## 8. 覆盖优先级

审计后的缺口按 P0/P1/P2 处理。

| 优先级 | 定义 | 处理方式 |
| --- | --- | --- |
| P0 | 影响核心知识完整性或工程可用性 | 必须生成文章或 reference patch |
| P1 | 不影响主线，但会影响查阅和排错 | 建议补入 reference patch |
| P2 | 原书示例局部符号、平台 helper、低价值表项 | 可在审计中记录，不强行正文展开 |

P0 通常包括：

- 语言扩展和 atomic memory model。
- 浮点和数值函数。
- Graph node/update/capture 限制。
- Unified Memory advice/prefetch/range attributes。
- Driver JIT/link/module/entry point。
- VMM/IPC/interop 系统边界。
- 原书关键图片和关键代码路径。

## 9. 验证清单

每轮生成后必须做验证。

### 9.1 文件与结构

- Markdown 文件数量符合预期。
- 总索引能链接到所有文章。
- 每篇文章有统一骨架。
- reference patch 已挂入阅读路径。

### 9.2 Markdown 健康度

- 代码块闭合。
- Mermaid 代码块使用带 `mermaid` info string 的 fenced code block。
- 图片链接存在。
- 表格语法可读。
- 没有旧模板残留。

旧模板残留关键词示例：

```text
本段主旨
原文核对
中文教材正文
讲解：
<details>
```

### 9.3 覆盖性

- 原书纳入章节全部有映射。
- 原书图片覆盖统计清楚。
- 原书关键代码路径有迁入或解释。
- API/token 扫描后弱覆盖项有记录。
- P0 缺口必须关闭。

### 9.4 可读性抽查

抽查每个模块至少一篇文章，检查：

- 开头是否像技术文章。
- 是否存在知识堆砌。
- 是否有工程判断。
- 是否解释了代码。
- 是否有图或表帮助理解。
- 是否能不看原书也读懂主线。

严格知识库模式下额外检查：

- 知识库是否包含 `Q&A` 和 `更新记录` 区块。
- 搜索确认包含关键规则或实际内容：`P5-Fig2`、`待确认`、`Q&A`、`更新记录`、`2000px`、`代码对比`、`【扩展】`。
- 代码片段是否有功能、关键语句、输入输出、依赖 API/库和出处。
- 拆分代码是否已还原，或明确说明无法还原的原因。

## 10. 文件命名建议

推荐目录结构：

```text
<Book Name>/
  original.pdf
  pdf_to_markdown_plan.md
  markdown_output/
    chapters/
    assets/images/
  course_materials/
    00-course-plan.md
    source_coverage_checklist.md
    v3_source_coverage_audit.md
    v3/
      00-index.md
      01-module-name/
        01-article-name.md
        02-article-name.md
      02-module-name/
        ...
```

命名规则：

- 模块使用两位数字编号。
- 文章使用两位数字编号。
- 文件名用英文短语，避免中文路径导致工具兼容问题。
- 标题可以是中文。
- 审计文件独立保留。
- 不建议删除中间版本，除非确认不再需要追溯。

严格知识库模式下建议额外增加：

```text
  knowledge_base/
    00-knowledge-index.md
    01-domain-overview.md
    02-topic-tree.md
    03-code-and-config-index.md
    04-qa-and-update-log.md
```

其中 `knowledge_base/` 是可选目录。如果项目规模较小，也可以只用一份 `knowledge_base.md`。

## 11. Definition of Done

一套书籍技术文章材料完成，至少满足：

- 有原书解析层。
- 有顶层课程/模块规划。
- 有可读主线文章。
- 有 reference patch 或等价补全区。
- 有总索引。
- 有覆盖审计报告。
- 关键图片全部纳入或明确排除理由。
- P0 缺口关闭。
- Markdown 结构、代码块、图片链接校验通过。
- 读者可以按索引从头读，也可以按 reference patch 查细节。

严格知识库模式启用时，还应满足：

- 有逐页出处标注。
- 关键图、表、代码都有统一编号。
- 模糊内容集中进入 `未明确点`。
- 用户确认后的问答已回填到 `Q&A` 和对应章节。
- `更新记录` 能追踪每次知识库变更。
- 博客或单篇文章只作为派生产物，不覆盖主线教材和 reference patch。

## 12. 后续处理新书时的执行顺序

处理下一本技术书时，按这个顺序推进：

1. 解析 PDF，生成章节 Markdown、图片、代码块。
2. 做章节目录和内容密度扫描。
3. 写 `pdf_to_markdown_plan.md` 或等价解析计划。
4. 写 `00-course-plan.md`，确定模块和阅读路径。
5. 生成第一版主线文章。
6. 人工阅读抽查，修正“教案味”和“机械味”。
7. 做 `source_coverage_audit.md`。
8. 按 P0/P1 生成 reference patch。
9. 更新总索引。
10. 运行验证脚本。
11. 更新审计报告中的最终状态。

如果启用严格知识库模式，在第 1 步后插入：

- 生成 `知识库：[文档名称]`。
- 建立逐页出处编号。
- 汇总代码/配置、图片/表格、未明确点。
- 后续问答确认后持续回填 `Q&A` 和 `更新记录`。

## 13. 最重要的质量标准

最终材料要同时满足三件事：

| 标准 | 含义 |
| --- | --- |
| 读得下去 | 像一组认真写的中文技术文章，不像自动摘要 |
| 查得到 | API、宏、图、代码、限制条件能在索引或补全区找到 |
| 对得上 | 能回到原书章节、图片、代码和 token 做覆盖核对 |

如果三者冲突，优先级是：

1. 主线文章保持可读。
2. reference patch 保证细节。
3. 审计报告记录差异。

不要为了“看起来完整”毁掉正文阅读体验，也不要为了“读起来轻松”隐瞒覆盖缺口。
